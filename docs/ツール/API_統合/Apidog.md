# Apidog

## 概要

Apidog は、API 設計、モック、テスト、ドキュメント公開を一体で扱える API 開発プラットフォームである。OpenAPI を中心に、設計から検証までの作業を1つのワークスペースに集約できる。

## 料金

| プラン | 内容 |
|------|------|
| Free | 個人利用向けの無料枠あり |
| 有料プラン | チーム管理、高度な権限設定、運用機能を拡張 |

## 主な特徴

| 項目 | 内容 |
|------|------|
| 統合ワークスペース | 設計、テスト、モック、ドキュメントを一元管理 |
| OpenAPI 対応 | OpenAPI / Swagger のインポート・エクスポート |
| モック生成 | 仕様からモックサーバーを素早く作成 |
| テスト運用 | 手動実行から自動実行まで対応 |
| チーム協業 | 共有、レビュー、権限管理に対応 |

## 主な機能

### API 設計機能

| 機能 | 説明 |
|------|------|
| API エディタ | エンドポイント、パラメータ、レスポンスを定義 |
| スキーマ管理 | 再利用可能なデータモデルを管理 |
| バリデーション | 仕様記述の整合を確認 |
| 仕様入出力 | OpenAPI / Postman 形式を相互変換 |

### テスト・自動化機能

| 機能 | 説明 |
|------|------|
| API 実行 | リクエスト送信とレスポンス検証 |
| テストケース | シナリオ単位で API テストを管理 |
| 環境変数 | 開発・検証・本番の設定を分離 |
| CLI 実行 | CI/CD からテストを自動実行 |

### モック・ドキュメント機能

| 機能 | 説明 |
|------|------|
| モックサーバー | 仕様ベースでモック応答を提供 |
| サンプルデータ | 例データを使ったレスポンス確認 |
| ドキュメント公開 | 仕様から API ドキュメントを生成・共有 |
| 変更共有 | 更新内容をチームで確認しやすい |

## インストールとセットアップ

公式URL:
- [Apidog 公式サイト](https://apidog.com/)
- [Download](https://apidog.com/download/)
- [Help Center](https://apidog.com/help/)
- [Docs](https://apidog.com/docs/getting-started/)

セットアップの要点:
1. デスクトップ版またはWeb版でワークスペースを作成する。
2. 新規プロジェクトを作成し、API設計対象を決める。
3. 既存仕様がある場合は OpenAPI / Postman をインポートする。
4. 環境変数と認証情報を設定し、テスト実行準備を行う。

## 基本的な使い方

1. エンドポイントとスキーマを定義して API 仕様を作成する。
2. 仕様からモックサーバーを起動し、先行開発で利用する。
3. テストケースを作成し、レスポンスとアサーションを確認する。
4. ドキュメントを公開し、利用者・実装者と認識を揃える。
5. CLI でテストを実行し、CI/CD に品質ゲートを組み込む。

最小実行例:
- 設計入力: OpenAPI 仕様または GUI で API 定義
- 出力: モック URL、テスト結果、公開ドキュメント

## メリット

- API 設計からテストまでを単一ツールで運用しやすい。
- OpenAPI 中心の資産管理で、仕様共有と変更追跡を進めやすい。
- モックと自動テストにより、実装前検証を早めやすい。

## デメリット

- 高度機能を使うほど運用ルール整備が必要になる。
- 既存ツール群からの移行には学習コストが発生する。
- プラン差分により使える機能が変わるため事前確認が必要。

## 他ツールとの比較

| ツール | 主な対象 | 特徴 |
|------|------|------|
| Apidog | API 設計〜テスト統合 | 設計・モック・テストを一体運用しやすい |
| Postman | API テスト中心 | 手動検証とコレクション運用に強い |
| Stoplight Studio | API 設計中心 | OpenAPI 設計とレビューに強い |
| Swagger/OpenAPI | 仕様標準 | 仕様定義とエコシステム連携が強い |

## ベストプラクティス

### 1. 仕様を先に確定

- 実装前に OpenAPI とサンプルレスポンスを揃える。
- 命名規約とエラーフォーマットを共通化する。

### 2. 環境変数と認証情報を分離

- 環境ごとに URL とトークンを分ける。
- 機密値は CI secrets で注入する。

### 3. 自動実行を標準化

- 主要 API を回帰テストとして定義する。
- PR 時点で CLI テストを実行して品質を担保する。
