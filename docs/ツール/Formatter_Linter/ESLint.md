# ESLint

## 概要

ESLint は JavaScript/TypeScript 向けの静的解析ツールである。コード品質や規約違反を自動検知し、修正支援も行えるため、フロントエンド開発の品質維持に広く使われる。

## 料金

| 区分 | 内容 |
|------|------|
| ESLint 本体 | 無料（OSS） |
| 補足 | 実行環境・拡張プラグイン運用コストは別途 |

## 主な特徴

| 項目 | 内容 |
|------|------|
| 柔軟なルール | プロジェクト規約に合わせて調整可能 |
| 自動修正 | 一部違反は `--fix` で修正可能 |
| プラグイン豊富 | React/Vue/TypeScript などに対応 |
| 連携性 | IDE、pre-commit、CI で運用しやすい |
| 標準化 | チームで同じ品質基準を適用可能 |

## 主な機能

### 解析機能

| 機能 | 説明 |
|------|------|
| ルール検証 | バグ/可読性/規約違反を検出 |
| TypeScript 対応 | TS 専用ルール適用 |
| プラグイン適用 | フレームワーク別に拡張 |
| ignore 設定 | 対象外ファイルを明示管理 |

### 修正機能

| 機能 | 説明 |
|------|------|
| 自動修正 | 対応ルールを一括修正 |
| 警告/エラー制御 | ルール重要度を調整 |
| Flat Config | 新しい設定形式で一元管理 |
| 共有設定 | チーム共通ルールとして配布可能 |

### 運用機能

| 機能 | 説明 |
|------|------|
| IDE 連携 | 保存時チェック・修正 |
| pre-commit | コミット前検証で品質担保 |
| CI 連携 | PR 段階で逸脱を検出 |
| レビュー支援 | 指摘の機械化でレビュー効率化 |

## インストールとセットアップ

公式URL:
- [ESLint 公式](https://eslint.org/)
- [ESLint Docs](https://eslint.org/docs/latest/)

セットアップ手順:
1. `eslint` と必要プラグインを導入する。
2. プロジェクトに設定ファイルを作成する。
3. npm scripts と CI に lint 実行を組み込む。

## 基本的な使い方

1. まず推奨ルールセットで開始する。
2. ルール衝突が出る箇所を整理し、最小限のカスタマイズを行う。
3. 日常開発では保存時 lint、PR では必須 lint を運用する。
4. 重大度が高いルールから段階的に厳格化する。

最小コマンド:
- 解析: `npx eslint .`
- 自動修正: `npx eslint . --fix`

## メリット

- バグ予防と規約統一を同時に進められる
- 自動修正で開発効率を維持しやすい
- 多くのフレームワークに対応できる
- CI 連携で品質ゲート化しやすい

## デメリット

- ルール設計を誤るとノイズが増える
- プラグイン依存が増えると保守が重くなる
- 既存コード導入時に初期対応コストがかかる

## 他ツールとの比較

| ツール | 主な用途 | 特徴 |
|------|------|------|
| ESLint | JS/TS 静的解析 | ルール柔軟性が高い |
| Biome | JS/TS 品質統合 | lint+format を統合しやすい |
| Prettier | JS/TS 整形 | コード整形に特化 |
| TypeScript Compiler | 型検証 | 型安全性確認に特化 |

## ベストプラクティス

### 1. ルールは段階導入

- まず推奨セットで開始する
- ノイズを減らしながら厳格化する

### 2. 役割分担を明確化

- 整形は Prettier、品質は ESLint など責務を分ける
- 競合ルールを避ける

### 3. CI で必須化

- PR 時に lint エラーをブロックする
- 例外は明示ルールで最小化する
