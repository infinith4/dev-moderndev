# EditorConfig

## 概要

EditorConfig はエディタ間でコードスタイル設定を共有するための仕組みである。インデント、改行、文字コードなどの基本ルールを統一し、環境差異による不要差分を減らせる。

## 料金

| 区分 | 内容 |
|------|------|
| EditorConfig 仕様 | 無料 |
| 補足 | 利用する IDE/拡張の費用は別途の場合あり |

## 主な特徴

| 項目 | 内容 |
|------|------|
| 軽量 | 単一ファイルで導入可能 |
| 互換性 | 多くの IDE/エディタが対応 |
| 差分削減 | 改行・インデント差異を抑制 |
| 設定継承 | ディレクトリ単位で適用範囲を制御 |
| 規約連携 | Linter/Formatter と併用しやすい |

## 主な機能

### スタイル設定機能

| 機能 | 説明 |
|------|------|
| インデント指定 | スペース/タブ、幅を定義 |
| 改行コード | LF/CRLF を統一 |
| 文字コード | UTF-8 などを指定 |
| 末尾改行 | ファイル終端改行を統一 |

### 適用制御機能

| 機能 | 説明 |
|------|------|
| パターン指定 | 拡張子ごとに設定分離 |
| ルート宣言 | 設定継承の起点を定義 |
| 部分上書き | サブディレクトリで局所設定 |
| 共通化 | 多言語プロジェクトで再利用 |

### 運用機能

| 機能 | 説明 |
|------|------|
| Git 管理 | 設定変更をレビュー可能 |
| 開発標準化 | 新規参加者も即適用可能 |
| 自動整形補助 | Formatter 連携で効果増大 |
| ルール明文化 | スタイル議論の削減に寄与 |

## インストールとセットアップ

公式URL:
- [EditorConfig 公式](https://editorconfig.org/)
- [EditorConfig Core](https://github.com/editorconfig/editorconfig)

セットアップ手順:
1. リポジトリ直下に `.editorconfig` を作成する。
2. 共通ルール（改行、インデント、文字コード）を定義する。
3. IDE 側で EditorConfig サポートを有効化する。

## 基本的な使い方

1. まず全言語共通の最小ルールを設定する。
2. 言語ごとに必要な差分だけ追加する。
3. 既存コードを整形してベースラインを揃える。
4. 変更時はフォーマッタ設定との整合を確認する。

最小設定例:
- ファイル名: `.editorconfig`

## メリット

- 不要なスタイル差分を減らせる
- 環境依存の揺れを抑えやすい
- 導入が簡単で効果が出やすい
- チーム規約の土台として使いやすい

## デメリット

- 高度な整形は単体では対応できない
- IDE によってサポート差がある
- ルール設計が不十分だと効果が限定的

## 他ツールとの比較

| ツール | 主な用途 | 特徴 |
|------|------|------|
| EditorConfig | 基本スタイル統一 | 軽量で導入しやすい |
| Prettier | 詳細整形 | 自動整形を強力に実施 |
| dotnet format | .NET 整形 | .NET 規約適用に強い |
| Black | Python 整形 | Python 向け一貫整形 |

## ベストプラクティス

### 1. 共通ルールを先に固定

- インデント・改行・文字コードを最優先で統一する
- 言語別ルールは最小追加に留める

### 2. 他ツールと整合

- Prettier/Linter と衝突しない設定にする
- 設定責務を明確に分ける

### 3. 継続レビュー

- ルール変更は PR で合意する
- 新規言語追加時に設定漏れを防ぐ

## まとめ

1. **基盤力** : EditorConfig は環境差異による無駄な差分を抑える基本基盤。
2. **安定性** : 共通ルールを先に固定すると運用が安定する。
3. **連携性** : Formatter/Linter と役割分担して使うと効果が高い。
