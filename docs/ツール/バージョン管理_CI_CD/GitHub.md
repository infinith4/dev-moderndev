# GitHub

## 概要

GitHub は Git リポジトリを中心に、コードレビュー、課題管理、CI/CD 連携を提供する開発プラットフォームである。Pull Request ベースの開発フローと豊富な連携機能により、チーム開発の標準基盤として利用しやすい。

## 料金

| プラン | 内容 |
|------|------|
| Free | 個人/小規模向けの基本機能 |
| Team | チーム運用向けの管理機能を拡張 |
| Enterprise | 大規模組織向けの監査・統制機能 |

## 主な特徴

| 項目 | 内容 |
|------|------|
| Gitホスティング | リポジトリ管理、ブランチ保護、アクセス制御 |
| PR中心開発 | レビュー、コメント、承認フローを一元化 |
| 課題管理 | Issues/Projects でタスク進捗を可視化 |
| CI/CD統合 | GitHub Actions で自動化を実装しやすい |
| セキュリティ機能 | Dependabot、Code Scanning などを提供 |
| エコシステム | 多数の外部ツール連携・Marketplace が利用可能 |

| 機能 | 説明 |

## インストールとセットアップ

公式URL:
- [GitHub](https://github.com/)
- [GitHub Docs](https://docs.github.com/ja)
- [GitHub Pricing](https://github.com/pricing)

セットアップの要点:
1. 組織または個人アカウントでリポジトリを作成する。
2. `main` ブランチ保護と必須ステータスチェックを設定する。
3. PR テンプレートと Issue テンプレートを追加する。
4. 必要に応じて Actions Secrets と Environment を定義する。

## 基本的な使い方

1. ローカルでブランチを作成し、変更を push する。
2. Pull Request を作成し、レビューを依頼する。
3. CI 結果とレビューコメントをもとに修正する。
4. 承認後に `main` へマージする。
5. タグ付けしてリリースノートを管理する。

最小運用例:
- ルール: `main` は直接 push 禁止
- 必須: PR 1件以上の承認 + CI 成功

## メリット

- Pull Request 中心でレビュー運用を標準化しやすい。
- Actions を含む統合機能で導入コストを抑えやすい。
- 外部連携が豊富で既存開発基盤と組み合わせやすい。

## デメリット

- 高度な組織統制機能は上位プラン前提になりやすい。
- ルール設定を増やすと運用が複雑化しやすい。
- プラットフォーム依存が強くなりやすい。

## CI/CD での使用

GitHub Actions と組み合わせ、`push`/`pull_request` をトリガーにビルド、テスト、デプロイを自動化する運用が一般的である。Environment 保護ルールとレビュー承認を組み込むことで、本番デプロイ時の統制を強化できる。

## ベストプラクティス

### 1. 保護ルールを先に定義

- `main` は保護し、必須チェックを固定する。
- 管理者を含めた例外運用を最小化する。

### 2. PR品質を定型化

- PRテンプレートで目的・影響範囲・テスト結果を必須化する。
- CODEOWNERS でレビュー責任を明確にする。

### 3. セキュリティ運用を継続

- Dependabot と Security Alert の対応SLAを決める。
- Secrets のローテーションを定期運用に組み込む。
