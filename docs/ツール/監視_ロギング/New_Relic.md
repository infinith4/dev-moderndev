# New Relic

## 概要

New Relic はアプリケーション監視（APM）を中心とした統合オブザーバビリティプラットフォームである。アプリ性能、インフラ、ログ、トレースを横断的に可視化し、障害調査と性能改善を進めやすい。

## 料金

| 区分 | 内容 |
|------|------|
| Free | 無償枠あり |
| Standard/Pro/Enterprise | データ量・機能に応じた有料プラン |

## 主な特徴

| 項目 | 内容 |
|------|------|
| APM 強み | トランザクション性能分析に強い |
| 統合監視 | インフラ・ログ・トレースを統合可能 |
| SaaS 運用 | 環境保守負担が低い |
| 分析機能 | NRQL による柔軟な分析が可能 |
| 導入速度 | エージェント導入で早期可視化しやすい |

## 主な機能

### 監視機能

| 機能 | 説明 |
|------|------|
| APM | レイテンシ、エラー率、スループットを可視化 |
| インフラ監視 | ホスト・コンテナ指標を監視 |
| ログ管理 | ログ収集と分析を統合 |
| 分散トレース | 呼び出し経路と遅延要因を追跡 |

### 分析機能

| 機能 | 説明 |
|------|------|
| NRQL | カスタムクエリで詳細分析 |
| ダッシュボード | 監視指標を統合表示 |
| 異常検知 | 閾値・学習ベース通知を設定 |
| インシデント管理 | 通知先連携と優先度管理 |

### 運用機能

| 機能 | 説明 |
|------|------|
| チーム共有 | ダッシュボードとアラートを共有 |
| 権限制御 | 役割ベースのアクセス管理 |
| 外部連携 | Slack、PagerDuty、Webhook 連携 |
| SLA/SLO 運用 | サービス品質目標の監視に利用可能 |

## インストールとセットアップ

公式URL:
- [New Relic 公式](https://newrelic.com/)
- [New Relic Docs](https://docs.newrelic.com/)

セットアップ手順:
1. 監視対象システムを整理する。
2. 対象環境へエージェントを導入する。
3. 主要KPIダッシュボードとアラートを設定する。
4. 運用レビューで通知精度を調整する。

## 基本的な使い方

1. まず主要アプリの APM 指標を監視開始する。
2. 次にインフラ・ログ・トレースを統合し調査導線を作る。
3. SLO に紐づくアラートを優先設定する。
4. 月次で不要通知と過剰データを見直す。

## メリット

- APM とオブザーバビリティを一体運用しやすい
- SaaS で導入が速い
- 障害調査の初動を短縮しやすい
- 指標分析と共有がしやすい

## デメリット

- データ量に応じて費用が増加しやすい
- 導入範囲が広いほど運用設計が必要
- ベンダー依存度が高くなる

## 他ツールとの比較

| ツール | 主な用途 | 特徴 |
|------|------|------|
| New Relic | APM/統合監視SaaS | APM 分析と統合可視化に強い |
| Datadog | 統合監視SaaS | 多機能で運用導入しやすい |
| Grafana + OSS | 可視化中心 | OSS 柔軟性が高い |
| Dynatrace | APM/監視SaaS | 自動検知機能が強い |

## ベストプラクティス

### 1. 監視対象を優先順位付け

- 重要サービスから導入する
- 全面導入を急がない

### 2. KPI と通知を連動

- SLO に直結する指標を通知対象にする
- ノイズ通知を定期的に削減する

### 3. データコストを管理

- 収集範囲と保持期間を設計する
- 利用価値の低いデータを削減する
